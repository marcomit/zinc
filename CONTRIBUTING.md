# Contributing to Zinc

Thank you for your interest in Zinc! This document describes how to contribute effectively.

Zinc is a personal project maintained by a single owner. All contributions are welcome, but the project vision and final decisions rest with the maintainer.

---

## Setup

After cloning, install the pre-push hook so tests run before every push:

```bash
cp hooks/pre-push .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

## How to Contribute

### Reporting Bugs

Open a GitHub issue with:

- A minimal `.zn` file that reproduces the problem
- The compiler output or error message
- Expected vs. actual behavior
- Your LLVM version (`llvm-config --version`)

### Suggesting Features

Open a GitHub issue describing:

- The use case motivating the feature
- How it fits into the existing language design
- Any syntax or semantics you have in mind

Features that conflict with the language's goals (clarity, performance, C interop) are unlikely to be accepted, but discussion is welcome.

### Submitting Code

1. Fork the repository and create a branch from `main`.
2. Make your changes (see guidelines below).
3. Open a pull request with a clear description of what changed and why.

All PRs are reviewed by the maintainer. There is no SLA on review time.

---

## Code Style

- Follow the existing C style in each file — indentation, brace placement, naming conventions (do not use any formatter since the code is formatted manually).
- Keep functions focused and small.
- Avoid adding dependencies beyond `clang` and LLVM.
- Do not break the existing test files in `tests/`.
- When adding a language feature, add a corresponding test case in `tests/`.

---

## Use of LLMs

You are welcome to use LLMs to assist with contributions — for exploration, drafting, or understanding unfamiliar parts of the codebase.

However:

- **You are responsible for every line you submit.** AI-generated code must be reviewed, tested, and understood by you before opening a PR. Do not paste output blindly.
- **Correctness is your responsibility.** LLMs can produce plausible-looking but incorrect C or LLVM IR. Verify that generated code compiles, passes tests, and matches the compiler's existing behavior.
- **Do not inflate scope.** LLMs tend to over-engineer. Keep changes minimal and focused, as described in the code style section above.
- **Disclose AI assistance in your PR** if a significant portion of the code was generated by an LLM. A brief note is enough — this is not a disqualifier, just a transparency expectation.
